# Project Constitution

本项目（A 股分析工具）的根本大章，定义项目的核心理念、设计原则、技术决策指南和代码审查标准。

---

## 一、项目价值观和原则

### 1.1 核心理念

**让投资分析更简单、更可靠**

- **准确性优先**: 数据必须准确可靠，宁可延迟也不返回错误数据
- **用户友好**: 输出清晰易懂的分析报告，避免过度专业化术语
- **稳健至上**: 系统必须在数据源失败、API 超时时仍能优雅降级

### 1.2 设计原则

1. **分层架构 (Layered Architecture)**
   - 数据层 (data/)：负责数据获取和存储，不关心业务逻辑
   - 核心层 (core/)：纯粹的业务逻辑，无外部依赖
   - 服务层 (services/)：报告生成、通知等跨领域功能
   - 表现层 (main.py)：CLI 入口，负责编排和输出

2. **依赖倒置**
   - 高层模块不依赖低层模块，都依赖抽象
   - 数据源通过 `BaseDataSource` 抽象，可插拔替换

3. **失败隔离 (Failure Isolation)**
   - 单一模块失败不应导致整个系统崩溃
   - 情绪分析 AI 失败时降级到本地规则
   - 数据源失败时自动切换备用源

4. **显式优于隐式**
   - 函数参数显式声明，避免隐式全局变量
   - 异常类型明确，错误信息清晰

## 二、技术决策指南

### 2.1 技术选型标准

| 技术 | 选型理由 | 替代方案 |
|------|---------|---------|
| **Python 3.10+** | 类型注解语法更简洁 (| 联合类型)，生态成熟 | - |
| **pandas** | 时间序列数据处理事实标准 | numpy (过于底层) |
| **dataclass** | 比 namedtuple 更灵活，比 attrs 更简单 | attrs, pydantic |
| **logging** | 标准库，无额外依赖 | loguru (增加依赖) |
| **pytest** | 测试编写简单，插件生态丰富 | unittest (语法冗长) |
| **akshare** | 开源免费，A 股数据覆盖全面 | tushare (需要积分) |
| **FastAPI** | 轻量快速，自动文档，适合事件接收 | Flask (较老), aiohttp |
| **uvicorn** | 高性能 ASGI 服务器，支持 FastAPI | gunicorn (WSGI) |

### 2.2 架构决策记录 (ADR)

#### ADR-001: 使用分层架构而非模块化

**背景**: 项目初期代码耦合严重，测试困难

**决策**: 采用 data → core → services → main 四层架构

**后果**:
- ✅ 各层职责清晰，易于测试
- ⚠️ 增加文件数量和导航复杂度

#### ADR-002: 数据源使用策略模式

**背景**: 单一数据源不稳定，需要多源容错

**决策**: `DataDownloader` 维护数据源列表，按优先级尝试

**后果**:
- ✅ 系统容错能力大幅提升
- ✅ 新增数据源只需实现 `BaseDataSource` 接口
- ⚠️ 错误日志需要追踪具体数据源

#### ADR-003: AI 分析可降级

**背景**: Claude API 可能不可用，用户可能无 API Key

**决策**: 情绪分析支持 AI 和本地规则两种模式，AI 失败时自动降级

**后果**:
- ✅ 无 API Key 用户仍可使用核心功能
- ⚠️ 本地规则准确度低于 AI

#### ADR-004: 飞书机器人双向互动功能

**背景**: 用户希望在飞书群聊中通过@机器人直接触发股票分析，实现双向互动

**决策**:
- 使用 FastAPI 创建轻量 HTTP 服务器接收飞书事件推送
- 复用现有分析引擎 (`StockAnalyzer`) 和报告服务 (`ReportService`)
- 通过飞书开放平台 API 发送交互式卡片消息
- 支持内网穿透工具进行本地开发测试

**后果**:
- ✅ 支持群聊中@机器人触发分析
- ✅ 复用现有代码，新增模块集中在 `app/bot/`
- ⚠️ 需要配置飞书开放平台和应用凭证
- ⚠️ 事件订阅 URL 需要公网可访问（本地开发需内网穿透）

**新增文件**:
- `app/bot/` - 飞书机器人模块
- `docs/FEISHU_BOT_SETUP.md` - 配置指南文档

**新增依赖**:
- `fastapi>=0.100.0` - Web 框架
- `uvicorn[standard]>=0.23.0` - ASGI 服务器

### 2.3 何时引入新技术

**准入标准** (必须全部满足):
1. 解决现有方案无法解决的问题
2. 学习成本低于 2 小时
3. 有稳定的维护记录和文档
4. 不增加过多依赖体积 (>5MB 需审批)

**禁止引入**:
- 处于 alpha/beta 阶段的库
- 最后更新时间超过 1 年的库
- 需要复杂配置的框架

## 三、代码审查清单

### 3.1 代码风格检查

- [ ] 遵循 PEP 8 规范
- [ ] 行宽不超过 100 字符
- [ ] 导入顺序：标准库 → 第三方库 → 本地模块
- [ ] 使用 4 空格缩进，不使用 Tab
- [ ] 无未使用的 import 和变量

### 3.2 类型注解检查

- [ ] 所有公共函数有类型注解
- [ ] 函数返回值有类型标注
- [ ] 复杂类型使用 typing 模块 (Optional, List, Dict)
- [ ] 无裸 `Any` 类型使用（确实无法确定类型时需注释说明）

### 3.3 文档字符串检查

- [ ] 公共函数有 docstring
- [ ] docstring 包含 Args 和 Returns 说明
- [ ] 类有类级别的文档说明

### 3.4 错误处理检查

- [ ] 无裸 `except:` 语句
- [ ] 使用项目定义的异常类（见 `exceptions.py`）
- [ ] 异常信息清晰，包含上下文数据
- [ ] 日志使用 `logger` 而非 `print`
- [ ] 外部 API 调用有超时设置

### 3.5 测试检查

- [ ] 新功能有对应的单元测试
- [ ] 测试覆盖正常路径和异常路径
- [ ] 测试不依赖外部 API（使用 mock 或模拟数据）
- [ ] 所有测试可通过 (`pytest tests/`)

### 3.6 架构检查

- [ ] 无跨层调用（如 core 直接调用 data 内部实现）
- [ ] 数据访问通过 `DataRepository` 或 `DataDownloader`
- [ ] 配置通过 `get_config()` 获取，无直接实例化 `Config`
- [ ] 新增数据源继承 `BaseDataSource` 并实现全部抽象方法

### 3.7 安全检查

- [ ] 无硬编码的 API Key 或敏感信息
- [ ] 用户输入有验证（如股票代码格式）
- [ ] 文件操作无路径遍历风险
- [ ] 网络请求有超时和重试限制

## 四、开发工作流

### 4.1 提交规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 类型**:
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具配置

**示例**:
```
feat(sentiment): 添加新闻去重功能

- 实现基于相似度的新闻去重算法
- 支持自定义相似度阈值

Closes #42
```

### 4.2 分支策略

```
main          - 稳定版本，可随时发布
  ↑
dev           - 开发集成分支
  ↑
feat/xxx      - 功能分支，从 dev 检出
fix/xxx       - 修复分支，从 dev 检出
```

### 4.3 发布流程

1. 在 dev 分支完成功能开发和测试
2. 创建 release 分支进行最终验证
3. 合并到 main 并打 tag
4. 更新 CHANGELOG.md

### 5.1 关键文件索引

| 文件 | 说明 |
|------|------|

| `app/utils/logger.py` | 日志配置 |

### 5.2 常见问题

**Q: 如何添加新的技术指标？**

A: 在 `app/core/technical.py` 中添加计算函数，然后在 `_generate_signals` 中生成对应信号。

**Q: 如何添加新的数据源？**

A: 继承 `BaseDataSource`，实现三个抽象方法，然后在 `DataDownloader` 中注册。

**Q: 如何修改报告模板？**

A: 编辑 `app/services/report.py` 中的 `_build_*_section` 方法。
